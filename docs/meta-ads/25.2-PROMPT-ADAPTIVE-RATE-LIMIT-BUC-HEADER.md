# Fix Prompt 25.2 — Rate-limit thích ứng theo `X-Business-Use-Case-Usage` (xử lý gốc lỗi 80004)

> **Bối cảnh**: Prompt `25-PROMPT-PARALLEL-AD-CREATION.md` (song song hoá tạo ad/creative) và `25.1-PROMPT-FIX-PARALLEL-ORPHAN-DUPLICATE.md` (chống orphan/duplicate) đã được implement trong `backend/MediationPro.Infrastructure/Services/MetaAds/MetaCampaignExecutionService.cs`.
>
> Prompt **25.2** này nâng cấp cơ chế retry rate-limit từ **"retry mù" (blind exponential backoff)** sang **thích ứng theo usage header thực tế của Meta**. Mục tiêu: xử lý gốc lỗi `OAuthException code 80004 / subcode 2446079` ("too many calls to this ad-account") thay vì cứ retry đôi delay cố định.
>
> Tài liệu độc lập, đủ ngữ cảnh để agent khác thực thi mà không cần lịch sử hội thoại.

---

## 1. Vấn đề hiện tại (retry mù)

Cơ chế retry hiện có trong `ExecuteCreateWithRetryAsync` (~dòng 892) và `CalculateRetryDelayMs` (~dòng 980):

```csharp
// ExecuteCreateWithRetryAsync
if (attempt >= maxAttempts || !IsRateLimitFailure(lastResult))
    throw new MetaCreateApiException(...);
var delayMs = CalculateRetryDelayMs(attempt);   // base * 2^(attempt-1) + jitter, cap 60s
await Task.Delay(delayMs, cancellationToken);
```

```csharp
// IsRateLimitFailure (~971): nhận diện rate-limit nhưng KHÔNG đọc usage
private static bool IsRateLimitFailure(MetaCreateApiCallResult result)
{
    if (result.HttpStatusCode == 429) return true;
    var code = (result.MetaError?.Code ?? string.Empty).Trim();
    return code is "4" or "17" or "32" or "80004";
}
```

**Điểm yếu:**
1. **Không đọc header usage** Meta trả về mỗi response (`X-Business-Use-Case-Usage`, `X-Ad-Account-Usage`). Trong `CallMetaCreateApiAsync` (~922), `response.Headers` đang bị bỏ qua hoàn toàn.
2. **Backoff cố định mù**: với 80004 (rate-limit cấp ad-account, cửa sổ trượt **1 giờ**), `estimated_time_to_regain_access` có thể là **vài phút → hàng chục phút**. Exponential cap 60s là **quá ngắn** ⇒ retry sớm → lại 80004 → đốt thêm quota → vòng lặp vô ích, cuối cùng fail.
3. **Không hãm song song**: gate `SemaphoreSlim(_adCreationMaxParallelism)` chạy hết tốc lực kể cả khi usage đã 95%. Burst làm thủng quota nhanh hơn.

## 2. Nền tảng (đã xác minh từ tài liệu Meta)

Meta trả các header usage trong **mọi** response Marketing API:

### `X-Business-Use-Case-Usage` (JSON, key theo business-id, value là mảng theo type)
```json
{
  "<business-id>": [{
    "type": "ads_management",
    "call_count": 95,                       // % quota call đã dùng
    "total_cputime": 20,                    // % CPU
    "total_time": 40,                       // % wall-clock
    "estimated_time_to_regain_access": 12,  // PHÚT phải đợi khi đã bị khoá (0 nếu chưa khoá)
    "ads_api_access_tier": "standard_access"
  }]
}
```

### `X-Ad-Account-Usage` (JSON)
```json
{ "acc_id_util_pct": 9.67, "reset_time_duration": 3600, "ads_api_access_tier": "standard_access" }
```

**Quan trọng về đơn vị:**
- `estimated_time_to_regain_access`: **PHÚT** → đổi sang ms khi delay.
- `reset_time_duration`: **giây**.
- `call_count` / `total_cputime` / `total_time` / `acc_id_util_pct`: **phần trăm** (0–100). Khi `total_cputime` hoặc `total_time` chạm 100 cũng bị throttle dù `call_count` chưa đầy.

## 3. Yêu cầu triển khai

### 3.1. Đọc & parse header usage trong `CallMetaCreateApiAsync`
Sau khi có `response` (RestSharp `RestResponse`, ~dòng 929), đọc 2 header (tên header **không phân biệt hoa thường**):

```csharp
var bucHeader = response.Headers?
    .FirstOrDefault(h => string.Equals(h.Name, "x-business-use-case-usage", StringComparison.OrdinalIgnoreCase))?.Value?.ToString();
var accHeader = response.Headers?
    .FirstOrDefault(h => string.Equals(h.Name, "x-ad-account-usage", StringComparison.OrdinalIgnoreCase))?.Value?.ToString();
```

Parse thành một record mới, ví dụ:
```csharp
private sealed record MetaUsageSnapshot(
    double? CallCountPct,            // từ BUC ads_management.call_count, fallback acc_id_util_pct
    double? TotalCpuTimePct,
    double? TotalTimePct,
    int? EstimatedTimeToRegainMinutes,
    string? AccessTier);             // development_access | standard_access
```

Quy tắc parse:
- `X-Business-Use-Case-Usage`: duyệt mọi business-id, lấy phần tử mảng có `type == "ads_management"` (nếu không có type này thì lấy phần tử đầu). Đọc `call_count`, `total_cputime`, `total_time`, `estimated_time_to_regain_access`, `ads_api_access_tier`.
- Nếu thiếu BUC, fallback `X-Ad-Account-Usage`: `acc_id_util_pct` → CallCountPct, `ads_api_access_tier`.
- Parse phòng thủ: header có thể vắng/sai định dạng ⇒ trả `MetaUsageSnapshot` rỗng (toàn null), **không được ném**. Bọc `try/catch` quanh parse, log Debug khi lỗi.

### 3.2. Mang usage vào `MetaCreateApiCallResult`
Thêm field `MetaUsageSnapshot? Usage` vào record `MetaCreateApiCallResult` (~2226) và set ở cả `Succeeded(...)` lẫn `Failed(...)`. Lý do: cả response thành công lẫn lỗi đều mang header usage; ta cần đọc usage **kể cả khi call thành công** để hãm proactive.

### 3.3. Backoff theo `estimated_time_to_regain_access` (thay retry mù)
Trong `ExecuteCreateWithRetryAsync`, khi gặp rate-limit failure và **còn lượt retry**:
- Nếu `Usage.EstimatedTimeToRegainMinutes` có giá trị `> 0`:
  `waitMs = EstimatedTimeToRegainMinutes * 60_000` (+ jitter nhỏ).
- Ngược lại (header vắng) → fallback `CalculateRetryDelayMs(attempt)` như cũ.
- **Cap an toàn**: `waitMs = Math.Min(waitMs, _adCreationRateLimitMaxWaitMs)`.
  - Nếu thời gian chờ ước tính **vượt cap** (vd Meta báo phải đợi 30 phút mà cap là 2 phút): **KHÔNG ngủ dài trong request** → fail fast ngay với thông điệp rõ ("rate limited, regain in ~N minutes; aborting so caller can retry later"), ném `MetaCreateApiException` để dừng. Người dùng retry sau (idempotency từ 25.1 đảm bảo không tạo trùng).
- Log Warning kèm `call_count`, `total_time`, `total_cputime`, `estimated_time_to_regain_access`, `access_tier`, `waitMs`.

> Lý do fail-fast khi vượt cap: 80004 ở cấp ad-account dùng cửa sổ **1 giờ**; giữ 1 HTTP request treo nhiều phút là phản mô hình (timeout, giữ kết nối, giữ DbContext scope). Tốt hơn là dừng sạch, đã persist xong những gì thành công (nhờ 25.1), để retry sau.

### 3.4. Hãm song song thích ứng (adaptive concurrency / pacing)
Thêm một **trạng thái rate-limit chia sẻ giữa các variant task trong cùng một lần execute** (thread-safe vì chạy song song). Tối thiểu cần một cơ chế **pause + pacing** dùng chung:

Thiết kế MVP đề xuất — class `MetaAdAccountThrottle` (instance tạo trong method điều phối song song, truyền vào các task):
- Trường `volatile`/lock-guarded: `DateTimeOffset _pauseUntilUtc`, `double _lastCallCountPct`, ...
- `Task WaitTurnAsync(CancellationToken ct)` — gọi **trước mỗi** API call (đầu `RunVariantApiAsync`, hoặc đầu `CallMetaCreateApiAsync`):
  - Nếu `now < _pauseUntilUtc` → `await Task.Delay(_pauseUntilUtc - now, ct)`.
  - Nếu usage gần đây vượt **soft threshold** (`_lastCallCountPct >= _adCreationRateLimitSoftThresholdPct` hoặc `total_time/total_cputime >= threshold`) → chèn **pacing delay** nhỏ (vd 500–1500ms, có jitter) để giãn nhịp.
- `void Observe(MetaUsageSnapshot usage)` — gọi **sau mỗi** response:
  - Cập nhật `_lastCallCountPct` v.v.
  - Nếu là throttle (80004 / 429) và có `EstimatedTimeToRegainMinutes` → đặt `_pauseUntilUtc = now + min(estimate, cap)` để **mọi task khác cùng dừng**, tránh các task song song còn lại tiếp tục đốt quota.
- **Thread-safety bắt buộc**: `Observe`/`WaitTurnAsync` chạy đồng thời từ nhiều task ⇒ dùng `lock` cho các trường, hoặc `Interlocked`/`volatile` hợp lý. KHÔNG chạm `_dbContext` trong throttle (chỉ state trong bộ nhớ).

Hiệu ứng: khi một task dính 80004, `_pauseUntilUtc` được set ⇒ các task song song còn lại tự khựng đúng khoảng Meta yêu cầu thay vì tiếp tục bắn request và làm tình hình tệ hơn. Đây chính là phần "hạ song song thích ứng".

> Ghi chú phạm vi: MVP chỉ cần throttle **trong một lần execute** (một campaign request). Nâng cao (tùy chọn, KHÔNG bắt buộc): nâng `MetaAdAccountThrottle` thành **singleton keyed theo ad-account id** để chia sẻ trạng thái rate-limit **xuyên nhiều request đồng thời** trên cùng account — hiệu quả hơn nhưng phức tạp về vòng đời DI; chỉ làm nếu dễ và an toàn.

### 3.5. Cấu hình mới (đọc trong constructor như các key sẵn có)
- `MetaAds:AdCreationRateLimitMaxWaitMs` (int, default **120000**, clamp `[1000, 600000]`) — cap thời gian chờ theo `estimated_time_to_regain_access`; vượt → fail-fast.
- `MetaAds:AdCreationRateLimitSoftThresholdPct` (int, default **80**, clamp `[1, 100]`) — ngưỡng usage % để bắt đầu pacing/hãm.
- (tùy chọn) `MetaAds:AdCreationPacingDelayMs` (int, default **800**, clamp `[0, 10000]`) — pacing delay khi vượt soft threshold.

## 7. Gợi ý chữ ký (định hướng, không bắt buộc nguyên văn)

```csharp
private sealed record MetaUsageSnapshot(
    double? CallCountPct, double? TotalCpuTimePct, double? TotalTimePct,
    int? EstimatedTimeToRegainMinutes, string? AccessTier)
{
    public static readonly MetaUsageSnapshot Empty = new(null, null, null, null, null);
    public bool IsThrottled(int softThresholdPct) =>
        (CallCountPct ?? 0) >= softThresholdPct ||
        (TotalTimePct ?? 0) >= softThresholdPct ||
        (TotalCpuTimePct ?? 0) >= softThresholdPct;
}

private sealed class MetaAdAccountThrottle
{
    private readonly int _softThresholdPct;
    private readonly int _pacingDelayMs;
    private readonly object _sync = new();
    private DateTimeOffset _pauseUntilUtc = DateTimeOffset.MinValue;
    private bool _nearLimit;

    public MetaAdAccountThrottle(int softThresholdPct, int pacingDelayMs) { ... }

    public async Task WaitTurnAsync(CancellationToken ct)
    {
        DateTimeOffset pauseUntil; bool nearLimit;
        lock (_sync) { pauseUntil = _pauseUntilUtc; nearLimit = _nearLimit; }
        var now = DateTimeOffset.UtcNow;
        if (now < pauseUntil) await Task.Delay(pauseUntil - now, ct);
        else if (nearLimit) await Task.Delay(_pacingDelayMs + Random.Shared.Next(0, 250), ct);
    }

    public void Observe(MetaUsageSnapshot usage, bool isThrottleError, int capMs)
    {
        lock (_sync)
        {
            _nearLimit = usage.IsThrottled(_softThresholdPct);
            if (isThrottleError && usage.EstimatedTimeToRegainMinutes is > 0)
            {
                var waitMs = Math.Min((long)usage.EstimatedTimeToRegainMinutes.Value * 60_000, capMs);
                var candidate = DateTimeOffset.UtcNow.AddMilliseconds(waitMs);
                if (candidate > _pauseUntilUtc) _pauseUntilUtc = candidate;
            }
        }
    }
}
```

`ExecuteCreateWithRetryAsync` (sửa nhánh delay):
```csharp
var usage = lastResult.Usage ?? MetaUsageSnapshot.Empty;
int delayMs;
if (usage.EstimatedTimeToRegainMinutes is > 0)
{
    var wanted = (long)usage.EstimatedTimeToRegainMinutes.Value * 60_000;
    if (wanted > _adCreationRateLimitMaxWaitMs)
        throw new MetaCreateApiException($"Ad-account rate limited; regain in ~{usage.EstimatedTimeToRegainMinutes}m. Aborting for later retry.", lastResult);
    delayMs = (int)Math.Min(wanted + Random.Shared.Next(0, 1000), _adCreationRateLimitMaxWaitMs);
}
else
{
    delayMs = CalculateRetryDelayMs(attempt);
}
```

## 8. Vì sao bản này là "xử lý gốc"

`80004 / 2446079` là rate-limit cấp **ad-account, cửa sổ 1 giờ**. Retry mù với delay 60s chỉ làm Meta cộng thêm call vào quota đang cạn ⇒ kéo dài việc bị khoá. Đọc `estimated_time_to_regain_access` cho ta **đúng khoảng thời gian Meta yêu cầu**, còn `call_count`/`total_time` cho ta tín hiệu **sớm** để hãm trước khi thủng. Kết hợp với idempotency (25.1), hệ thống vừa không tạo trùng vừa không tự bắn thủng quota — đúng cách mà tài liệu Meta khuyến nghị (monitor BUC header thay vì thử-sai).
