# Prompt (FIX): Sửa Nguồn Data & Tách EU/APAC Cho Ad Transparency (Beneficiary/Payer)

> **Bối cảnh**: Feature ở `docs/meta-ads/45-PROMPT-AD-TRANSPARENCY-BENEFICIARY-PAYER.md` **ĐÃ được implement**. Tuy nhiên bản implement có **lỗi nguồn data & gộp sai 2 cơ chế**. Prompt này CHỈ mô tả phần **cần sửa** — không làm lại từ đầu. Agent phải đọc code hiện có rồi chỉnh đúng các điểm dưới.

## Vấn đề của bản implement hiện tại
1. **Sai endpoint nguồn dropdown**: dùng `GET /act_<id>/dsa_recommendations` để lấy danh sách Advertiser. Endpoint này **chỉ phục vụ EU DSA** và **luôn trả rỗng** với regulated APAC (Singapore/Taiwan) → dropdown không có data. Đây là nguyên nhân "không call API ra được Advertiser".
2. **Gộp sai EU và APAC** thành một cơ chế. Thực tế Meta tách **2 block độc lập**, dùng **2 field khác nhau, 2 kiểu nhập khác nhau**.

## Sự thật đã verify (đọc ngược ad set thật trên Graph API v24.0)
Đọc 1 ad set đã cấu hình đầy đủ trả về:
```jsonc
"regional_regulated_categories": ["TAIWAN_UNIVERSAL","SINGAPORE_UNIVERSAL","AUSTRALIA_FINSERV","TAIWAN_FINSERV","THAILAND_UNIVERSAL","BRAZIL_REGULATION"],
"regional_regulation_identities": {
    "taiwan_universal_beneficiary":    "915965694080940",   // BM ID
    "taiwan_universal_payer":          "915965694080940",
    "taiwan_finserv_beneficiary":      "915965694080940",
    "taiwan_finserv_payer":            "915965694080940",
    "singapore_universal_beneficiary": "1394085102311573",  // BM ID khác
    "singapore_universal_payer":       "1394085102311573",
    "universal_beneficiary":           "1394085102311573",
    "universal_payer":                 "1394085102311573"
},
"dsa_beneficiary": "Amobear",    // EU = TEXT (tên), KHÔNG phải ID
"dsa_payor": "Amobear"
```
Một ad set khác chỉ điền EU:
```jsonc
"regional_regulated_categories": ["SINGAPORE_UNIVERSAL"],
"dsa_beneficiary": "AMOBEAR PTE",   // text
"dsa_payor": "AMOBEAR PTE"
// KHÔNG có regional_regulation_identities → chưa chọn advertiser verified → ad bị giữ, Ads Manager hiện cảnh báo "4 tasks required"
```

**Kết luận đã xác nhận:**
- **Value của `regional_regulation_identities` = Business Manager (BM) ID** đã verify (người dùng đã confirm 2 ID trên là BM ID). KHÔNG phải Page ID, KHÔNG phải free text.
- **`dsa_beneficiary`/`dsa_payor` (EU) = chuỗi TEXT tên advertiser** do user gõ. KHÔNG phải ID, KHÔNG cần verify, KHÔNG cần dropdown.
- SG và TW là **2 BM ID khác nhau** trong **cùng 1 map** → 1 ad set khai nhiều location cùng lúc (không phải "1 ad set 1 quốc gia").
- Chỉ điền `dsa_*` text mà thiếu `regional_regulation_identities` thì ad **không deliver** ở SG/TW (Meta giữ ad). Muốn deliver phải có BM ID trong `regional_regulation_identities`.

## Mapping UI → API (2 block riêng, đúng như Ads Manager)
| Block UI | Field API | Kiểu nhập | Hiện khi |
|---|---|---|---|
| "Advertiser and payer details required for ads in the EU" | `dsa_beneficiary` / `dsa_payor` | **1 ô text tên** (+ toggle payer khác) | ad set target EU |
| "Ad transparency" (Singapore, Taiwan, Other locations) | `regional_regulation_identities` (map BM ID) | **Dropdown chọn Business** (+ toggle payer khác) | ad set target SG/TW/regulated APAC |

## Việc cần sửa

### FIX 1 — Bỏ `dsa_recommendations`, đổi nguồn dropdown sang danh sách Business
- Trong reference service đang gọi `dsa_recommendations` (file mới mà agent đã tạo, vd `MetaRegulationIdentityReferenceService` / controller `MetaReferenceController` action regulation-identities): **xóa call `dsa_recommendations`**.
- Thay bằng **list Business Manager** mà integration token truy cập được, trả `{ id, name, verificationStatus }`:
  - Graph: `GET /me/businesses?fields=id,name,verification_status` (dùng access token của integration đã resolve qua `MetaAuthManager` / `MetaAdAccountExecutionContextResolver` — mirror cách `MetaFacebookPageReferenceService` resolve context).
  - Nếu token là system-user gắn 1 business và `/me/businesses` rỗng, thử thêm `GET /<business_id>/owned_businesses` và `/client_businesses` (business_id lấy từ `adAccount.BusinessId` nếu có).
  - Lọc ưu tiên `verification_status` đã verify nhưng **vẫn trả cả list** (Ads Manager hiển thị mọi BM verified để user tự chọn; SG/TW eligibility là phần nội bộ Meta, public API không phân biệt được — đừng cố filter theo category).
- **Fallback seed config**: nếu token thiếu scope `business_management` (list rỗng/permission error), đọc danh sách từ config/DB seed (org-scoped). Seed tối thiểu cho hiện tại:
  - `1394085102311573` — AMOBEAR PTE. LTD (Singapore)
  - `915965694080940` — AMOBEAR PTE. LTD (Taiwan)
- Endpoint reference giữ nguyên đường dẫn FE đang gọi (đừng đổi contract FE nếu không cần); chỉ đổi phần lấy data bên trong.

### FIX 2 — Tách EU (text) khỏi APAC (BM ID) ở DTO/State
- Đảm bảo DTO ad set có **đồng thời**:
  - `DsaBeneficiary` (string, text), `DsaPayor` (string, text) — cho EU.
  - `RegionalRegulationIdentities` (map BM ID, các key `singapore_universal_*`, `taiwan_universal_*`, `taiwan_finserv_*`, `australia_finserv_*`, `india_finserv_*`, `universal_*`).
- Nếu bản implement đang nhét chung EU vào dropdown ID (hoặc nhét APAC vào text) → tách lại đúng kiểu. EU **không** dùng dropdown Business; APAC **không** dùng text tự do.

### FIX 3 — Execution payload đúng kiểu
- `BuildAdSetPayloadAsync` (`MetaCampaignExecutionService.cs`, sau block set `regional_regulated_categories`):
  - `dsa_beneficiary`/`dsa_payor`: gửi **đúng chuỗi text** từ DTO (đừng resolve qua ID).
  - `regional_regulation_identities`: gửi map **BM ID**, chỉ thêm key của category đang bật trong `regional_regulated_categories`. Payer rỗng → payer = beneficiary.
- Kiểm tra lại `MetaAdSetDraftValidationService.BuildAdSetPayloadAsync` đồng bộ.

### FIX 4 — UI 2 block riêng
- File `section-adset-audience.tsx` (hoặc nơi agent đã đặt block):
  - **Block EU**: 1 input text "Advertiser" + toggle "payer khác" → input text "Payer". Hiện khi target EU.
  - **Block Ad transparency**: dropdown **select-only** (options = list Business từ FIX 1, hiển thị `name` + verification badge, value = BM id) cho từng location (Singapore, Taiwan, Other/universal). Toggle "payer khác" → dropdown Payer riêng. SG/TW = required.
- Bỏ mọi chỗ FE đang đổ data EU vào dropdown ID hoặc ngược lại.

### FIX 5 — Validation
- Giữ rule: target SG/TW mà thiếu BM ID beneficiary tương ứng → chặn submit, message rõ ràng.
- EU: nếu target EU mà thiếu `DsaBeneficiary` text → cảnh báo/chặn theo policy dự án.

## Notes
- Public Marketing API **không** có endpoint liệt kê đúng "verified identity" như dropdown nội bộ Ads Manager. Nguồn khả dụng public gần nhất = list **Business** (`/me/businesses`) + `verification_status`. Đây là lựa chọn đúng; `dsa_recommendations` là sai hướng cho APAC.
- BM ID là source of truth cho `regional_regulation_identities`; KHÔNG map qua Page ID.
- Giữ terminology: UI "Advertiser/Payer"; API EU = `dsa_beneficiary/dsa_payor` (text); API APAC = `regional_regulation_identities` (BM id).
