## 1. Ground truth — shape reference từ OpenAPI v4 (đã verify)

Coi đây là **mẫu kỳ vọng** của response (dashboard gần như dùng cùng schema). Đối chiếu khi bắt được request thật; nếu lệch → ghi theo thực tế.

- Data: `GET /v4/analytics/charts/{chart_code}` ; metadata: `…/{chart_code}/meta`.
- `chart_code` gồm: `proceeds`, `refunds`, `refund-rate`, `active-subscriptions`, `new-subscriptions`, `trial-to-paid`, `users-overview`, `user-to-trial` (alias `user-to-trial-conversion`), …
- Params: `from`,`to` (Unix giây), `unit`∈`hour|day|week|month`, `environment`∈`0|1`, `currency`, `segmentation` (vd `product_id`), `max_series`, `filter[<attr>][]`.
- Response `V4AnalyticsChart`: `measure`∈`usd|count|percent`, `totalType`∈`sum|wavg`, `seriesRelation`∈`partsOfWhole|independent`, `segmentation` (nullable), `series[]`, `summarySeries`.
- `series[]`: `label` (= giá trị segment, vd product id), `total`, `totalPrev`, **`totalWeight`** (weight của `total`, dùng cho wavg), `data[]`.
- `data[]`: CHỈ `start_time` (Unix giây) + `value`. **KHÔNG có weight theo từng ngày.**

> 2 suy luận then chốt: (1) `totalWeight` là **của cả range đã query**, không theo ngày; (2) `totalType` **do API trả theo response**, không cố định = `sum`.

## 3. Việc cần làm

### 3.1 Bắt request thật cho 4 report (auth = cookie)
Với mỗi chart, ghi vào Appendix:
- **Host + path đầy đủ + method** (xác nhận là `api.qonversion.io/v4/analytics/charts/...` hay endpoint nội bộ `dash.qonversion.io/...`).
- **Cách auth thật**: header `Cookie` (so với `AddDashboardHeaders` hiện có), có cần `origin`/`referer`/CSRF token không.
- Query params thật (chart_code, from/to, unit, segmentation, environment, currency, platform).
- **Response shape thật** (đối chiếu §1): có `series[].label`=product id, `total`, `totalWeight`, `data[]` không.

### 3.2 FIX 2 (P1) — Verify segmentation `product_id` NGAY TRONG UI
Không cần key: trên chart **New-User-to-Trial Conversion**, bật **Group by = Product**.
- Nếu UI hiển thị tách theo product → segmentation product **được hỗ trợ**; bắt request đó làm bằng chứng.
- Nếu UI **không** cho group theo product (chart users không hỗ trợ) → **STOP**: report #2 "by product" bất khả thi, ghi rõ đầu doc và báo cần bàn scope với PO.
- Làm tương tự cho `active-subscriptions`, `trial-to-paid`, `refunds`/`refund-rate`.

### 3.3 FIX 1 (P1) — Sửa công thức numerator theo ngày (sai)
A.6 Câu 1 / A.8 đang ghi `numerator = value_ngày × totalWeight` → **sai** (value là rate ngày, totalWeight là của cả range). Thay bằng:
- API không trả num/den, cũng không có weight theo ngày.
- Chỉ ở **mức series/range** mới có `total` (wavg) + `totalWeight` (≈ denominator cả range); numerator range ≈ `ROUND(total × totalWeight)`.
- Chiến lược crawl đúng (khuyến nghị **a**):
  - **(a) Per-day pull**: gọi API `from/to` = đúng từng ngày → mỗi ngày 1 series có `total`+`totalWeight` riêng → lưu theo ngày `numerator/denominator` đúng → gộp range = `Σnum/Σden`.
  - **(b) Per-range pull**: gọi đúng range dashboard cần (today/yesterday/last7), lưu ở mức range; đơn giản hơn nhưng không tái gộp.

### 3.4 FIX 3 (P2) — Active Subscriptions
Bỏ khẳng định "total = Σ daily_snapshot (sum)". Active subs là metric **tồn kho**, theo Qonversion là **số cuối kỳ**. ⇒ "trong range" của PO nhiều khả năng = điểm cuối `data[-1].value` (verify bằng `totalType` thật trong response). Giữ note cần chốt PO, diễn đạt là giả định-cần-verify.

### 3.5 FIX 4 (P2) — API v1
A.1 "API 2 v1" chưa có bằng chứng → đánh dấu **"(CHƯA VERIFY — bỏ qua)"** hoặc xoá.

### 3.6 FIX 5 (P3 — đổi theo hướng crawl) — Lật khuyến nghị nguồn
A.6 Câu 4 / A.1 đang khuyến nghị "dùng v4 Secret Key, bỏ cookie-crawl". **Lật lại**: nguồn chính thức là **cookie-crawl** (vì không có key). Ghi rõ: tái dùng `QonversionAccount.DashboardCookie` + `DashboardAccountUid` + pattern `AddDashboardHeaders`; v4 spec chỉ là reference shape. Rủi ro cookie hết hạn → cùng cơ chế renew như crawler raw-data hiện có.

### 3.7 FIX 6 (P3) — Hợp nhất §2 ↔ §4.1 ↔ Appendix
- §2 "QUY TẮC VÀNG Σnum/Σden": sửa cho khớp — API trả `total`+`totalWeight`; num/den chỉ suy ở mức range theo chiến lược pull (a)/(b).
- §4.1 schema: diễn giải cột theo dữ liệu thật — `metric_value` (= `value`/`total`), `total_weight`, `total_type`(`sum|wavg`), `measure`(`usd|count|percent`), `series_label`(product id); `numerator/denominator` chỉ điền khi dùng per-day (3.3a).
- §3/§4 doc 08: đổi mô tả pipeline cho khớp crawl (cookie) thay vì Bearer key.
