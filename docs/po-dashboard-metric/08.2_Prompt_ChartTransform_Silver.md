## 2. Vấn đề map app (ĐÃ điều tra — chốt cách làm)

Bronze chỉ có `qon_project_key` + `platform`, **KHÔNG có** admob_app_id. Dashboard (8.3) query theo **admob app id**, nên 8.2 phải map `(qon_project_key, platform) → admob_app_id`.

- Bảng `config.qonversion_project_mapping` (`nexus_app_id, qon_project_key, qon_app_store_id, platform, ...`) **đã tồn tại nhưng RỖNG** và **không có routine populate** (chỉ có DDL + mention trong comment). Đây là bảng được thiết kế đúng cho việc này.
- Nguồn map thật = **PostgreSQL `apps`**: `qonversion_params.projectKey` + `platform` → `app_id` (admob). Lưu ý: `apps.qonversion_params` là **text chứa JSON** (không phải jsonb) → cast `qonversion_params::jsonb->>'projectKey'` khi query Postgres, hoặc parse bằng EF/JsonDocument.
- `apps.platform` = `'ANDROID'`/`'IOS'` (chữ hoa) ≠ bronze `'Android'`/`'iOS'` → **chuẩn hoá khi JOIN** (so sánh không phân biệt hoa thường).

**Chốt cách làm (2 phần):**
1. **Populate `config.qonversion_project_mapping`** từ Postgres `apps` (sync nhỏ, idempotent) — vừa unblock 8.2, vừa đúng thiết kế (event pipeline cũng tham chiếu bảng này).
2. **bronze→silver bằng SQL thuần** JOIN `config.qonversion_project_mapping` trên `(qon_project_key, platform)`.

> Nếu vì lý do nào đó không muốn dùng `config.qonversion_project_mapping`: phương án thay thế là transform job đọc dict `(projectKey,platform)→app_id` từ EF rồi chạy INSERT…SELECT từng project (inject admob_app_id). Nhưng **ưu tiên** phương án populate config table (sạch hơn, StarRocks-native).

## 3. Bảng đích — `silver.qonversion_product_metrics_daily`

> ⚠️ **StarRocks: cột KEY phải đứng ĐẦU bảng, đúng thứ tự KEY** (Slice 8.1 từng fail vì đặt key ở giữa). Dùng **UNIQUE KEY** để re-transform = upsert idempotent.

```
UNIQUE KEY (report_date, admob_app_id, platform, product_id, metric_key)  -- 5 cột này PHẢI là 5 cột đầu, đúng thứ tự
report_date    DATE,
admob_app_id   VARCHAR(255),
platform       VARCHAR(16),
product_id     VARCHAR(255),     -- = product_label raw (PO yêu cầu raw product id)
metric_key     VARCHAR(64),
-- value columns:
measure        VARCHAR(10),      -- count | usd | percent
metric_value   DECIMAL(20,6),    -- giá trị ngày (rate% nếu percent; count/usd nếu khác)
numerator      DECIMAL(20,6) NULL,   -- percent: ROUND(metric_value/100 * total_weight, 6); count/usd: NULL
denominator    DECIMAL(20,6) NULL,   -- percent: total_weight; count/usd: NULL
qon_project_key VARCHAR(64),     -- giữ để trace
updated_at     DATETIME
```
- Partition theo tháng `report_date` + dynamic_partition (mirror bronze 8.1).
- DISTRIBUTED BY HASH(admob_app_id) BUCKETS … (hoặc theo pattern silver hiện có).

## 4. Logic transform

### 4.1 Populate `config.qonversion_project_mapping` (idempotent)
Từ Postgres `apps` (where `qonversion_crawler=true` và projectKey không rỗng), với mỗi `(projectKey, platform→{ANDROID→Android, IOS→iOS}, app_id)`:
- Upsert vào `config.qonversion_project_mapping`: `qon_project_key`, `platform` (đã chuẩn hoá `Android`/`iOS`), `nexus_app_id = app_id`, `qon_app_store_id` nếu có, `crawler_enabled=1`, `is_active=1`.
- Dedup: 1 dòng / `(qon_project_key, platform)`. Nếu nhiều app cùng project+platform → lấy 1 (vd app_id nhỏ nhất, **đồng nhất với cách crawler 8.1 dedup work item**).
- Thực hiện trong transform job (EF đọc Postgres → Stream Load/INSERT StarRocks), hoặc method riêng `SyncQonversionProjectMappingAsync`.

### 4.2 bronze → silver (SQL thuần)
```sql
-- pseudo, điều chỉnh theo style RunQonversion*Async hiện có
DELETE FROM silver.qonversion_product_metrics_daily
WHERE report_date BETWEEN @start AND @end;   -- (hoặc dựa UNIQUE KEY upsert, chọn 1 cơ chế nhất quán)

INSERT INTO silver.qonversion_product_metrics_daily
WITH dedup AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY report_date, qon_project_key, platform, chart_code, product_label
      ORDER BY ingested_at DESC
    ) AS rn
  FROM bronze.qonversion_chart_metrics_raw
  WHERE report_date BETWEEN @start AND @end
)
SELECT
  d.report_date,
  m.nexus_app_id            AS admob_app_id,
  d.platform,
  d.product_label           AS product_id,
  d.metric_key,
  d.measure,
  d.metric_value,
  CASE WHEN d.measure='percent' AND d.total_weight IS NOT NULL
       THEN ROUND(d.metric_value/100 * d.total_weight, 6) ELSE NULL END AS numerator,
  CASE WHEN d.measure='percent' THEN d.total_weight ELSE NULL END        AS denominator,
  d.qon_project_key,
  NOW() AS updated_at
FROM dedup d
JOIN config.qonversion_project_mapping m
  ON LOWER(TRIM(m.qon_project_key)) = LOWER(TRIM(d.qon_project_key))
 AND LOWER(TRIM(m.platform))        = LOWER(TRIM(d.platform))
WHERE d.rn = 1
  AND TRIM(COALESCE(m.nexus_app_id,'')) <> '';
```

### 4.3 Quy tắc (đã chốt từ data thật — A.6/A.7)
- **numerator = ROUND(metric_value/100 × total_weight)** cho percent (vì metric_value là %). **KHÔNG** dùng `metric_value × total_weight`.
- **denominator = total_weight** (percent); NULL cho count/usd.
- **active-subscriptions**: silver lưu **giá trị ngày** như mọi count khác. Việc "trong range = end-of-period (lấy ngày cuối)" là quyết định ở **8.3 provider**, KHÔNG xử lý ở 8.2.
- Bỏ qua bronze rows không map được app (m.nexus_app_id rỗng) — log số lượng bị bỏ.
